# Task Scheduler 仕様書

## 1. 概要
Google Tasks（ToDoリスト）とGoogle Calendar（カレンダー）を双方向に連携させ、AI (Gemini) を活用してタスクの自動スケジューリングを行うPythonツールの仕様です。

## 2. アーキテクチャ構成
*   **実行環境:** ローカル環境 (Windows)
*   **UI形態:** 簡易GUIアプリケーション (Flet または Tkinter)
    *   操作: ボタン押下によるステップ実行（振り分け開始 -> 実行開始）。
*   **開発言語:** Python 3.x
*   **API連携:**
    *   Google Calendar API
    *   Google Tasks API
    *   Gemini API (`google-generativeai`, 個別モデルとして `gemini-2.5-pro` 等を利用)
*   **構成方針:**
    *   Google Apps Script (GAS) は使用せず、Pythonから直接各種APIを操作する。
    *   Google Maps API は使用せず、移動時間の算出は「過去の履歴」または「Geminiによる推測」を用いる。
    *   **GUI:** Flet または Tkinter を使用した簡易デスクトップアプリとして実装する（ボタンによるステップ実行）。
    *   **コード記述:** 生成するPythonのコードには、後からの保守性を高めるために詳細なコメントを追記する。

## 3. 機能詳細

### 3.0. 対象範囲とリスト運用
*   **対象リスト:** タイトルが **`■`** で始まるタスクリストのみを本ツールの処理対象とする（例: `■仕事`, `■プライベート`）。
*   **非表示タスクリスト:** タイトルが **`■■`** で始まるタスクリストは作業用として用い、非表示タスクリストとする。
*   **Inbox:** タスクリスト `■メモ` をInbox（入り口）として扱う。
*   **除外:** `■` が付いていないリストは読み込み・操作の対象外とする。

### 3.1. 同期ロジック (Synchronize)
タスクからカレンダーへの一方向の同期を行います。

*   **Calendar → Tasks (カレンダーからタスクへ)**
    *   **同期しない:** カレンダーからタスクへの同期はいかなる場合でも行わない。カレンダー側で作成・変更・削除された予定はタスクに一切反映させない。
*   **Tasks → Calendar (タスクからカレンダーへ)**
    *   未スケジュールのタスクを分析し、カレンダーの空き時間に自動登録する。
    *   **重複防止:**
        *   一度カレンダーに登録したタスクは、タスクタイトルを `【予定済】元のタスク名` に変更して視認性を高める。
        *   システム的な二重登録防止として、タスクの「メモ欄 (Notes)」末尾に `[Ref:EventID:xxxxx]` の形式でカレンダーイベントIDを記録する。
    *   **予定の取り消し（タスク側のみ）:**
        *   一度カレンダーに登録されたタスク（`【予定済】` やメモ欄にIDがある状態）について、登録をなかったことにしたい場合は、ユーザーがタスク側でこれらの目印（タイトルの`【予定済】`の削除、メモ欄のID削除）を手動で外すことで「未登録」ステータスに戻すことができる。
        *   この操作を行っても、**カレンダー側で作成された予定は削除・変更されずに放置される**（手動で削除する必要がある場合はユーザーがカレンダー上で操作する）。
        *   **【テスト用機能】元に戻すボタン:** アプリ上の「元に戻す」ボタンは、上記の「タスク側の目印削除」を一括で自動実行するテスト用機能である。これもあくまでGoogle Task側のデータのみを振り分け前の状態に戻すものであり、カレンダー上のデータはいじらない。

### 3.2. タスク分析と分割 (Analysis & Breakdown)
Gemini APIを使用してタスクの内容を分析します。

*   **分析項目:** 重要度、緊急度、所要時間、場所、コンテキスト。
*   **タスクの自動分割:**
    *   所要時間が長い、または工程が複数あるタスクは、Geminiが自動的に**サブタスク**に分割する。
    *   **処理:** 親タスクは「コンテナ（入れ物）」扱いとしてスケジューリング対象外とし、分割された**子タスク**個々をスケジューリング対象とする。親タスク名は `【分割済】...` 等に変更する。
        *   ※親タスクと子タスクの完了ステータスの連動は行わない（独立して管理する）。
    *   **過剰分割の防止（ユーザー確認）:**
        *   1つのタスクが **n個以上（例: 5個以上）** の子タスクに分割された場合は、自動でのカレンダー登録処理を保留する。
        *   ユーザーに分割の正常性（適切に分割されているか等）を確認させた後、ユーザーが承認（またはGUI等で実行指示）を行ってからカレンダーへの登録処理へ進む。

        *   ※親タスクと子タスクの完了ステータスの連動は行わない（独立して管理する）。

### 3.3. タスクの振り分け (Organization)
*   **タイミング:** アプリ起動直後に実行 (「■メモー＞」ボタン等の実行トリガー)。
*   **処理:** Inbox (`■メモ`) にある複数のタスクを**AIへ一括（まとめて）渡し**、分析および最適な `■` 付きリストの判定をバッチ処理で行った後、各リストへ順次移動させる。
    *   振り分けの実行中は、進行状況（どのタスクがどのリストへ移動されたかなどのログ）をアプリのUI画面上へ逐次表示する。
    *   処理完了時には、完了を通知するダイアログを表示する。
*   **未分類:** 適切な移動先が見つからない場合は、`■メモ` にそのまま残す。
*   **修正:** 誤ったリストに移動された場合は、ユーザーが**Google Tasks公式アプリ/Web**上で手動修正する。本ツールは再読み込み機能を提供する。

### 3.4. 自動スケジューリング (Scheduling)
*   **優先順位とAIによる順序最適化:**
    *   従来はタスクの期限(Due Date)を最優先としてそのままの順序で処理を行っていたが、各タスクリスト（例: `■仕事`）ごとに、AI (Gemini) が**リスト内の全未スケジュールタスク群を俯瞰し、最適な実行順序を検討・並び替えた上で**カレンダーに登録する仕様とする。
    *   本最適化の検討単位は「リスト内」に限定し、リストをまたいだ順序の検討は行わない。
*   **所要時間の明示的指定:** タスク名の末尾に全角・半角問わず `：5H` や `：30m` の形式で記載することで、スケジュール登録時の時間幅として最優先で採用する。
    *   **書式:** `[タスク名] + [:または：] + [数値] + [H,h,M,m,Ｈ,ｈ,Ｍ,ｍ]` (全角半角・大文字小文字は問わない)。
    *   **適用条件:** 時間指定は**タスク名の最後尾のみ**を判定対象とし、名称の中間にある場合はテキストとして扱う。コロンの後に数値が未記載（または書式外）の場合は時間指定が無いものとみなす。
*   **空き枠探索と登録可能時間の制限:** 
    *   カレンダーの `Free/Busy` 情報を取得し、空いている時間にパズル的にタスクを割り当てる。
    *   **時間帯制限:** アプリケーション上のGUIから「登録開始時間（デフォルト06時）」と「登録終了時間（デフォルト22時）」を指定し、その時間帯の範囲内でのみタスクをスケジュールする。
    *   **深夜帯の回避:** 終了時間から翌日の開始時間まで（例：22:00〜翌06:00）はシステム的に「架空の予定（Busy）」としてブロックし、タスクの完了時刻が終了時間（22時など）をはみ出す場合は、自動的に翌日の開始時間（06時など）以降へ先送りされる。
    *   **日またぎの設定不可:** 設定は同日内（開始時間 ＜ 終了時間）のみをサポートし、夜勤のようないわゆる「日付をまたぐ」設定対象外（手動でカレンダー登録）とする。
    *   スケジュール登録中は、進行状況（どの予定を追加しているか等のログ）をアプリのメッセージウィンドウ（UI）へ逐次表示する。
    *   **カレンダー表示色:** 自動スケジューリングによってカレンダーに登録された予定は、手動で登録した予定と区別しやすくするため、標準で「クリーム色/淡い黄色（Banana）」で表示される。
    *   **予定タイトルの接頭辞:** 自動スケジューリングによってカレンダーに登録された予定であることが一目でわかるように、タイトルの先頭には常に「◆」が付与される。
*   **移動時間の考慮:**
    *   場所の移動が発生する場合（AIが場所を推定）、前後の予定との間に移動時間を確保する。
    *   移動時間が15分以上と予測される場合、カレンダー上に独立した「移動イベント」を作成する。
    *   移動時間の長さは、過去の同一区間の履歴を参照するか、履歴がない場合はGeminiに概算を出させる。
*   **制限事項:**
    *   スケジュール対象期間は最大「2週間先」までとする。
    *   当日の空きがない場合は翌日以降に繰り越す。

## 4. データ構造と管理
*   **ID紐付け:** `state_manager.py` 等を用いて、タスクIDとカレンダーイベントIDの紐付けを管理・永続化する。
*   **設定管理:** `.env` ファイルを利用してAPIキーなど環境依存の機密情報を管理する。`config.py` にはスケジュール期間や除外リストなどのアプリ共通設定を集約する。

## 5. 運用フロー
1.  ユーザーが `■メモ` にタスクを登録。
2.  スクリプト/アプリを起動。
3.  **[自動] 振り分け処理**: ボタン（「■メモー＞」等）押下にて、`■メモ` から各 `■リスト` へタスクが移動される。（進捗はUIログで確認、完了時はダイアログ表示）
4.  **[手動] ユーザー確認**: ユーザーがタスクの振り分け結果を **Google Tasks公式アプリ等で確認・修正** する。
5.  **[手動] スケジュール実行**: ユーザーがGUI上の実行ボタンを押下。
6.  **[自動] 分析・スケジューリング**:
    *   タスクの分析・分割。
    *   Calendar の空き枠への登録。（進捗はUIログで逐次確認）
7.  GUI上で完了状態を確認。

## 6. クイックフィードバック・ワークフロー（開発・テスト用）
テスト中や運用中に見つかった仕様変更やバグは、チャットベースのバックログ運用を用いて効率的に収集・反映する。

1.  **フィードバックの起票:**
    *   ユーザーはチャット内で **「バックログへ」** というキーワードを文頭に付けて要望を送信する。
    *   AIがチャットの文脈から「仕様変更や修正案として残すべき」と判断した場合も、ユーザーに提案して起票する。
2.  **バックログの蓄積:**
    *   内容は `doc/feedback_backlog.md` に【日付】と共に追加される。
3.  **一括反映:**
    *   テストの一区切りなど、ユーザーからの「まとめて修正して」という指示をトリガーとして、AIがバックログの内容をコードに一括で反映させる。
