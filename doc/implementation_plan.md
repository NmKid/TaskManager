# 要件分析と実装計画

## 1. 要件のまとめ

`specification.md` に基づく主要な要件は以下の通りです。

*   **目的**: Google Tasks と Google Calendar の連携および Gemini による自動スケジューリングツール。
*   **構成**:
    *   Python 3.x (Windows ローカル)。
    *   API: Google Tasks, Google Calendar, Gemini (google-generativeai)。
    *   DB: ローカルファイル (state_manager) でID紐付けを管理。
    *   除外: GAS, Maps API は使用しない。
    *   **コード記述**: 生成するPythonのコードには、後からの保守性を高めるために**詳細なコメントを追記**する。
*   **同期 (Sync)**:
    *   **対象**: `■` で始まるリストのみ。`■メモ` がInbox。
    *   **非表示タスクリスト**: `■■` で始まるリストは作業用として非表示タスクリストとして扱う。
    *   **Cal -> Tasks**: **同期しない**。カレンダーの作成・変更・削除はタスクに一切反映しない。
    *   **Tasks -> Cal**: 未スケジュールタスクを分析し、カレンダーの空き枠に登録。
        *   タスク名を `【予定済】...` に変更。
        *   詳細に `[Ref:EventID:xxxxx]` を追記。
        *   **予定の取り消し**: ユーザーがタスク側で `【予定済】` やメモ欄のIDを手動で削除することで「未登録」ステータスに戻す。この際、カレンダー側の予定は放置（削除・変更しない）。
*   **振り分け (Organization)**:
    *   起動時に `■メモ` のタスクを各 `■リスト` へ移動。
    *   該当なしは `■メモ` に維持。
*   **分析 (Analysis & Breakdown)**:
    *   Gemini がタスクを分析（所要時間、重要度、場所）。
    *   複合タスクは「サブタスク」に分割。親タスクはコンテナ化（タイトルを `【分割済】...` に変更）。
    *   **過剰分割の防止**: 1つのタスクが `n個以上`（例: 5個以上）の子タスクに分割された場合は、自動でのカレンダー登録を保留し、**ユーザーに正常性を確認**させるフローを挟む。
*   **スケジューリング (Scheduling)**:
    *   優先度: 期限 > 文脈（重要度など）。
    *   移動時間: 15分以上なら「移動イベント」を作成（履歴 または Gemini推測）。

## 2. 懸念点への対応方針

仕様書の更新により、以前存在した以下の矛盾点は解決しました。

*   **同期ループのリスク**: 「Cal -> Tasks」の同期が廃止されたため、ツールによる無限ループの発生懸念は解消されました。
*   **カレンダー削除時の不整合**: カレンダー側の操作はタスクに影響を与えないというルールにより解消されました（不整合は許容）。
*   **予定の一段階解除**: ユーザーがタスクの名称やメモ欄から手動で目印を消すことで、再スケジューリングの対象となる運用ルールが確立しました。

## 3. やり残していること・未定義事項

実装に向けて詳細化が必要な項目です。

*   **[ToDo] GUIフレームワークの選定**:
    *   要件「簡易GUI」に基づき、モダンなUIが容易な **Flet** (Flutter for Python) または標準の **Tkinter** を採用する。
    *   機能: 「振り分け実行ボタン」「スケジューリング実行ボタン」「ログ表示エリア」。さらに、**タスク過剰分割時の「確認・承認UI」** が必要となる。
    *   ※タスクそのものの編集・並べ替え機能は**実装しない**（ユーザーはGoogle Tasks公式側で操作する）。
*   **[ToDo] 移動イベントの定義**:
    *   移動イベントのタイトル形式（例: `移動 (A地点 -> B地点)`）や色（Color ID）の指定がありません。
*   **[ToDo] 履歴データの保存方法**:
    *   「過去の履歴」を参照する仕組みについて、簡単なJSONDBに「出発地:到着地:所要時間」を蓄積する形式で良いか確認が必要です。
*   **[ToDo] タスク変更の同期**:
    *   タスクの内容（メモやタイトル）が後から追加変更された場合、カレンダーイベントに再反映（Update）するかどうかが未定義です。最新仕様に基づけば「予定済」タスクはカレンダーを更新しない（再同期しない）方針が妥当と思われます。
*   **[ToDo] エラーハンドリング**:
    *   API制限（Quota）やネットワークエラー時の再試行ロジック。

## 4. 検証プラン (Verification Plan)

### 4.1. 手動検証ルート (GUI操作)
GUIアプリをベースとした検証を行います。
1.  **初期化**: Google Tasksの「■メモ」リストにテスト用のタスクを複数作成する。
    *   例1: 「1時間の打ち合わせ」
    *   例2: 「大規模プロジェクトの要件定義と設計とテスト」（必ず5個以上に分割されるようプロンプト等で調整）
    *   例3: 「■■作業用リスト」への移動テスト用タスク
2.  **実行 (振り分け)**: アプリを起動し、振り分け処理を実行。
    *   対象タスクが適切な `■` リスト（または `■■` リスト）に移動しているか確認。
3.  **実行 (スケジューリング)**: スケジュール実行ボタンを押下。
    *   タスクがカレンダーに登録され、名前が `【予定済】...` になり、メモ欄に `[Ref:EventID:...]` が追記されること。
    *   例2の大規模タスクが `n個以上` に分割された際、ただちにカレンダー登録されず、GUI上で「ユーザー承認待ち」のステータス/ダイアログが出現すること。
4.  **承認処理**: ユーザー承認待ちのタスクを「承認」し、カレンダーへ登録されることを確認。
5.  **取り消し処理の検証**:
    *   登録済みのタスク（例1）について、手動でGoogle Tasks上で `【予定済】` とメモ欄のIDを削除する。
    *   再度アプリでスケジュール実行を行い、同じタスクが再スケジュールされるか（未登録ステータスとして扱われるか）を確認する。
