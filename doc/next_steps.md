# 次回作業時のネクストステップ (Next Steps)

PC再起動後など、次回以降の作業を再開する手順と、デバッグ・検証に向けたタスクをまとめました。

## 1. 動作環境の準備（初回必須）

1.  **Google API 認証情報の配置**
    *   Google Cloud Console からダウンロードした OAuth クライアント ID の JSON ファイルを、以下のパスに配置してください。
    *   配置先: `d:\GH\TaskManager\config\credentials.json`
2.  **必要なライブラリのインストール**
    *   ターミナルを開き、依存ライブラリをインストールします。
    *   コマンド: `pip install flet google-api-python-client google-auth-httplib2 google-auth-oauthlib google-generativeai`
3.  **Gemini API キーの設定**
    *   Google AI Studio で取得した API キーを環境変数 `GEMINI_API_KEY` に設定します。
    *   Windows (PowerShell) の場合の実行例: `$env:GEMINI_API_KEY="あなたのAPIキー"`

## 2. 次回行うべきデバッグ・テスト作業

前回までの機能追加で、以下の機能が新たに実装/改善されています。

1.  **「元に戻す」ボタンのテスト**
    *   カレンダーにスケジュール実行済のタスク（`【予定済】` がついているもの）が存在する状態で、アプリの「元に戻す」ボタンを押下します。
    *   Google Tasks を確認し、タイトルから `【予定済】` が消え、メモ欄のID `[Ref:EventID:...]` が消去されていることを確認してください。
    *   ※この処理はTasks側のみを元に戻すものであり、カレンダーの予定は残ったままで正常です。
2.  **ログ出力の詳細化の確認**
    *   「スケジュール実行」時、アプリ下部のログに「『（タスク名）』をカレンダーへイベント登録中...」と表示されるか確認してください。
3.  **振り分け (Organization) 精度の確認**
    *   「■メモ」リストに様々なタスクを入れて「振り分け」ボタンを押下し、仕事とプライベート等でこれまでより正確に分類される傾向にあるか確認してください。
4.  **UI承認と登録取り消しのテスト（継続）**
    *   （前回から継続）巨大なタスクを作成して「過剰分割」と判定させ、GUIの「確認・承認が必要なタスク」欄に保留されるか確認。その後「内容を承認して登録」ボタンでのカレンダー登録を確認します。

## 3. 今後の拡張・検討事項

現状はテスト用機能（元に戻すボタンなど）も含めベースの実装が完了した段階です。テストを通して以下のカスタマイズをご検討ください。

*   **カレンダー空き時間の探索ロジックの強化:**
    *   現在は取得した Free/Busy データを元にした本格的なパズルロジック（予定と予定の隙間に埋め込む処理）までは組み切れておらず、順次詰め込む簡易動作に近い状態になっています。`scheduler.py` の `_schedule_single_task` を中心に空き時間の厳密な算出ロジックを強化していく必要があります。
*   **移動時間の考慮:**
    *   場所の移動が発生する場合の、移動時間の自動確保機能が未実装です。
*   **本番運用に向けた不要ボタンの整理:**
    *   「元に戻す」や「【予定済】削除」といったデバッグ用ボタンは、機能的に安定した段階でUIから隠すか削除することを検討します。
