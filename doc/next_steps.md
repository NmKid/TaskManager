# 次回作業時のネクストステップ (Next Steps)

PC再起動後など、次回以降の作業を再開する手順と、デバッグ・検証に向けたタスクをまとめました。

## 1. 動作環境の準備（初回必須）

1.  **Google API 認証情報の配置**
    *   Google Cloud Console からダウンロードした OAuth クライアント ID の JSON ファイルを、以下のパスに配置してください。
    *   配置先: `d:\GH\TaskManager\config\credentials.json`
2.  **必要なライブラリのインストール**
    *   ターミナルを開き、依存ライブラリをインストールします。
    *   コマンド: `pip install flet google-api-python-client google-auth-httplib2 google-auth-oauthlib google-generativeai`
3.  **Gemini API キーの設定**
    *   Google AI Studio で取得した API キーを環境変数 `GEMINI_API_KEY` に設定します。
    *   Windows (PowerShell) の場合の実行例: `$env:GEMINI_API_KEY="あなたのAPIキー"`

## 2. 次回行うべきデバッグ・テスト作業

### 動作確認テスト
以下の順序でアプリの動作を確認してください。

1.  **アプリの起動**
    *   `python src/main.py` を実行して、Flet の GUI が正常に立ち上がるか確認する。
    *   初回起動時はブラウザが開いて Google アカウントの認証（Tasks/Calendar）が求められるので、許可して認証を完了させる。
2.  **振り分け (Organization) のテスト**
    *   Google Tasks の公式画面を開き、**「■メモ」**(Inbox) リストに適当なタスクを作成する。
    *   アプリの `振り分け (Inbox -> Lists)` ボタンをクリックする。
    *   コンソール（またはGUIのログ）にエラーが出ないこと、およびタスクが適切な「■仕事」等のリストへ移動しているかを確認する。
3.  **スケジューリングと分割 (Scheduling) のテスト**
    *   Google Tasks のリスト内に、「1時間の会議」などの短いタスクと、「大規模プロジェクトの要件定義から実演まで」などの巨大なタスク（5個以上のサブタスクになることを期待）を作成する。
    *   アプリの `スケジュール実行 (Tasks -> Cal)` ボタンをクリックする。
    *   カレンダーに予定が入ることと、元タスクの名称が `【予定済】...` に変わることを確認する。
    *   巨大なタスクについては、GUIの「確認・承認が必要なタスク」の欄に保留状態として表示されることを確認する。
4.  **UI承認と登録取り消しのテスト**
    *   GUI上に保留された過剰分割タスクに対し、`内容を承認して登録` ボタンを押し、カレンダーへの登録が走るか確認する。
    *   手動で Tasks 側のタスク名を編集し、`【予定済】` とメモ欄のID `[Ref:EventID:...]` を消去した上で再度スケジュール実行ボタンを押し、対象のタスクが再処理されるか確認する。

## 3. 今後の拡張・検討事項

現状はベースとなる機能の実装が完了した段階です。テスト・デバッグを通して以下のカスタマイズをご検討ください。

*   **カレンダー空き時間の探索ロジックの強化:**
    *   現在は取得した Free/Busy データを元にした本格的なパズルロジック（予定と予定の隙間に埋め込む処理）までは組み切れておらず、順次詰め込む簡易動作に近い状態になっています。デバッグ後に `scheduler.py` の `_schedule_single_task` を中心に空き時間の厳密な算出ロジックを強化していく必要があります。
*   **移動時間の考慮:**
    *   場所の移動が発生する場合の、移動時間の自動確保機能が未実装です。
